\subsection[Typzugriff: tryCatchThrows]{Typzugriff: tryCatchThrows (Ordner \code{TypeAccess/tryCatchThrows})}
In der Klasse \code{A} des Features \feature{FeatureA} wird innerhalb eines try-catch-Blocks die statische Methode \code{bar()} der Klasse \code{B} des Features \feature{FeatureB} aufgerufen. Der try-catch-Block fängt eine Exception vom Typ \code{E}, die von der Methode \code{bar()} der Klasse \code{B} erzeugt und geworfen wird. Die Klasse \code{E} wird im Feature \feature{FeatureE} definiert.
\subsubsection{Fall 01: Das Feature \feature{FeatureE} ist im Feature-Modell als optional gekennzeichnet.}
Dadurch kann es passieren, dass in einer Variante die benötigte Klasse \code{E} nicht zur Verfügung steht, wenn das Feature \feature{FeatureE} nicht ausgewählt wurde.  
Dieser Fall löst insgesamt drei Fehlermeldungen aus:
\begin{Verbatim}[fontsize=\scriptsize, frame=single]
/TypeAccess/tryCatchThrows/01/FeatureB/B.java:3:
  Semantic Error: MAYBE dependency:
Feature FeatureB accesses the constructor
  public E() { ... }
of feature FeatureE.
Feature FeatureE may not be present in every valid selection.

/TypeAccess/tryCatchThrows/01/FeatureA/A.java:5:
  Semantic Error: 1 optional target:
Feature FeatureA accesses:
- the type
    (default package).E
  of feature FeatureE

/TypeAccess/tryCatchThrows/01/FeatureB/B.java:3
/TypeAccess/tryCatchThrows/01/FeatureB/B.java:2:
  Semantic Error: 1 optional target:
Feature FeatureB accesses:
- the type
    (default package).E
  of feature FeatureE
\end{Verbatim}
Um diese Fehler zu beseitigen, ist es ausreichend, zum Feature-Modell das Constraint \constraint{FeatureA $\implies$ FeatureE} hinzuzufügen.
Dadurch muss, wenn das Feature \feature{FeatureA} ausgewählt wurde, auch das Feature \feature{FeatureE} ausgewählt werden.
\subsubsection{Fall 02: Die Features \feature{FeatureB} und \feature{FeatureE} schließen sich im Feature-Modell gegenseitig aus.} Dadurch kann die Klasse \code{E} des Features \feature{FeatureE} nicht ausgewählt werden, wenn das Feature \feature{FeatureB} ausgewählt wird.
Dieser Fall löst insgesamt fünf Fehlermeldungen aus, wobei sich drei davon darauf beziehen, dass die Methode \code{bar()} der Klasse \code {B} und die Typen \code{B} und \code{E} aus Sicht der Klasse \code{A} nicht vorhanden sein könnten. Die anderen zwei Fehlermeldungen behandeln die Zugriffe der Klasse \code{B} auf die Klasse \code{E}:
\begin{Verbatim}[fontsize=\scriptsize, frame=single]
/TypeAccess/tryCatchThrows/02/FeatureA/A.java:4:
  Semantic Error: MAYBE dependency:
Feature FeatureA accesses the method
  public static void bar() throws E { ... }
of feature FeatureB.
Feature FeatureB may not be present in every valid selection.

/TypeAccess/tryCatchThrows/02/FeatureB/B.java:3:
  Semantic Error: NEVER dependency:
Feature FeatureB accesses the constructor
  public E() { ... }
of feature FeatureE.
Features FeatureB and FeatureE cannot both be present in a valid selection.

/TypeAccess/tryCatchThrows/02/FeatureA/A.java:5:
  Semantic Error: 1 optional target:
Feature FeatureA accesses:
- the type
    (default package).E
  of feature FeatureE

/TypeAccess/tryCatchThrows/02/FeatureA/A.java:4:
  Semantic Error: 1 optional target:
Feature FeatureA accesses:
- the type
    (default package).B
  of feature FeatureB

/TypeAccess/tryCatchThrows/02/FeatureB/B.java:3
/TypeAccess/tryCatchThrows/02/FeatureB/B.java:2:
  Semantic Error: 1 optional target:
Feature FeatureB accesses:
- the type
    (default package).E
  of feature FeatureE
\end{Verbatim}
Um diese Fehler zu beseitigen, ist es ausreichend, zum Feature-Modell die Constraints \constraint{FeatureA $\implies$ FeatureB} und \constraint{FeatureB $\implies$ FeatureE} hinzuzufügen.
Dadurch muss, wenn das Feature \feature{FeatureA} ausgewählt wurde, auch das Feature \feature{FeatureB} ausgewählt werden und wenn das Feature \feature{FeatureB} ausgewählt wurde, auch das Feature \feature{FeatureE}. Allerdings können dadurch die Features \feature{FeatureA} und \feature{FeatureB} in keiner Variante mehr ausgewählt werden. Eine Auswahl des Features \feature{FeatureB} würde im Feature-Modell zu einem Widerspruch führen, da das Feature \feature{FeatureB} zum einen das Feature \feature{FeatureE} impliziert, zum anderen aber mit diesem in gegenseitigem Ausschluss steht. Bei einer Auswahl des Features \feature{FeatureA} müsste auch das Feature \feature{FeatureB} ausgewählt werden.
\subsubsection{Fall 03: Die Klasse \code{E} wird identisch in den zwei optionalen Features \feature{FeatureE} und \feature{FeatureE2} definiert.} Für die Zugriffe aus der Klasse \code{B} heraus werden nun zwei Ziele angeboten, die aber beide nicht ausgewählt werden müssen. Dadurch kann es passieren, dass in einer Variante gar kein Ziel zur Verfügung steht, wenn beide Features nicht ausgewählt wurden:
\begin{Verbatim}[fontsize=\scriptsize, frame=single]
/TypeAccess/tryCatchThrows/03/FeatureB/B.java:3:
  Semantic Error: 2 optional targets 
  (there may be a valid selection where none of these targets is present):
Feature FeatureB accesses:
- the constructor
    public E() { ... }
  of feature FeatureE
- the constructor
    public E() { ... }
  of feature FeatureE2

/TypeAccess/tryCatchThrows/03/FeatureA/A.java:5:
  Semantic Error: 2 optional targets 
  (there may be a valid selection where none of these targets is present):
Feature FeatureA accesses:
- the type
    (default package).E
  of feature FeatureE
- the type
    (default package).E
  of feature FeatureE2

/TypeAccess/tryCatchThrows/03/FeatureB/B.java:2
/TypeAccess/tryCatchThrows/03/FeatureB/B.java:3:
  Semantic Error: 2 optional targets 
  (there may be a valid selection where none of these targets is present):
Feature FeatureB accesses:
- the type
    (default package).E
  of feature FeatureE
- the type
    (default package).E
  of feature FeatureE2
\end{Verbatim}
Um diese Fehler zu beseitigen, ist es ausreichend, zum Feature-Modell das Constraint \constraint{$\lnot$ FeatureE $\iff$ FeatureE2} hinzuzufügen.
Dadurch muss immer genau ein Exemplar der Klasse \code{E} ausgewählt werden.
